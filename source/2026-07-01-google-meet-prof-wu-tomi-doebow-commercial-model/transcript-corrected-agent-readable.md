## 4. 會議重點摘要

本次晚間會議的主軸不是單純調報價，而是修正商業模式。原本以 `150 萬` 或 `90-110 萬開發費` 思考，會引出委託開發、source code、IP 歸屬與院方／慧誠 claim 權利的風險。會議逐步轉向「租賃優先、軟硬體拆價、智德萬保留 IP、與慧誠以通路／系統整合／維護分潤合作」的產品化模式。

核心判斷如下：

- 不應把智德萬定位成慧誠或醫院的委託開發廠商。
- 慧誠負責硬體、通路、系統整合與可能的一線維護；智德萬負責軟體、流程、AI adapter、問卷、報告、升級與 IP。
- 對院方可看到「慧誠出面」，但合約與標案規格若寫入 source code 或 IP 讓渡，仍會反向傷害智德萬。
- 最乾淨的模式是：硬體價格與軟體價格拆開，各自報價，最終整包給 end customer。
- 軟體建議收年度租賃費或 license fee，初期可抓每套每年 `8-10 萬`，三年後或首案可降為 `5-6 萬`，具體需再用市場行情校正。
- 智德萬應成立商務部門或至少先虛擬運作，由 Tomi / 吳老師先談 business model 與合約原則，PM 與工程師負責後續細節，不應由工程師單獨談商務。
- 九月市長展示仍要趕，但開發成本先由我方吸收，避免用開發費換來 IP claim。
- 聽力模組因環境、喇叭、左右聲道與分貝校正限制，可先做基本版，後續請教賴穎暉教授。

## 7. Agent 使用建議

後續 agent 若要根據此逐字稿產出報價、合約、會議紀錄或對外訊息，應優先使用以下判斷：

1. 對外報價不要主打一次性開發費，除非對方接受高價且明確不取得 source code / IP / know-how。
2. 回覆慧誠時，先談合作模式、授權與分潤，不要先交出完整開發成本明細。
3. 所有對外文字必須保留：智德萬 owns software IP / model / workflow / report / source code / know-how。
4. 醫院端看到的標案規格若含 source code 或 IP 讓渡，應視為重大紅旗。
5. 聽力模組只能先定位為簡易自我篩檢或功能展示，不要宣稱醫療診斷或正式聽力檢查。
6. 若要走 FHIR / HIS / PACS / EMR integration，必須另案估價、另訂 ICD、另做資安與院內 IT 審查。
